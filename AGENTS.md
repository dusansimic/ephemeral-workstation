# AGENTS.md

Guidance for agents working in this repo.

## What this repo is

Terraform + Ansible to create and provision a **disposable Fedora 44 workstation** on Hetzner Cloud. Create → provision → use → destroy. Not long-lived infra; favor simplicity over hardening.

## Layout

```
terraform/   # provisions the VM, writes ansible/inventory/hosts.ini from a template
ansible/     # site.yml playbook + roles (repositories, packages, update, github_releases, user, directories, fish, ssh)
```

- Terraform state is **local** (`terraform.tfstate`, gitignored). No remote backend.
- `ansible/inventory/hosts.ini` (host IP + private-key path) and `ansible/inventory/group_vars/workstation.yml` (public key) are **generated by Terraform** from `var.ssh_public_key` / `var.ssh_private_key_file` (both gitignored). Don't hand-edit them. The SSH key material lives only in `terraform.tfvars`, never committed.
- The playbook (`ansible/site.yml`) owns `workstation_user`; `workstation_ssh_key` comes from the generated group_vars. No passwords anywhere — SSH-key login only, passwordless sudo.

## Conventions

- **Roles do one thing** (its name). Role execution order in `site.yml` is deliberate:
  `repositories → packages → update → github_releases → user → directories → fish → ssh` (docker group must exist before the user joins it; the user's home must exist before creating `~/Projects` and before installing fish plugins; SSH is hardened last, after the key is installed, to avoid lockout).
- **Config is data, not code.** Adding a repo / package / directory / fish plugin / GitHub-release tool = append to the role's
  `defaults/main.yml` list (`workstation_repositories`, `workstation_packages`, `workstation_directories`, `workstation_fish_plugins`, `workstation_github_releases`). Don't add new tasks for these.
- **Shared vars are intentionally not role-prefixed** — they use the `workstation_` prefix and are owned by the playbook. `ansible/.ansible-lint` skips `var-naming[no-role-prefix]` and `package-latest` (the `update` role upgrades to latest on purpose).

## Developing

Run these before committing (pre-commit runs them automatically):

```sh
# Terraform
cd terraform && terraform fmt -recursive && terraform validate

# Ansible
cd ansible && ansible-galaxy collection install -r requirements.yml   # first time
ansible-playbook --syntax-check site.yml
ansible-lint -c .ansible-lint .
```

Pre-commit (see `.pre-commit-config.yaml`):

```sh
pre-commit install          # once
pre-commit run --all-files  # run all hooks now
```

Full end-to-end verification (needs a real Hetzner token + SSH key) is documented in `README.md`.

## Committing

**When the `/caveman-commit` skill is run:**

- Generate a Conventional Commits message from the staged changes.
- **Do NOT add the `Co-Authored-By: Claude` trailer** or any "authored with agent" line. Commit with the generated message only.
- If there are **multiple unrelated changes** and the user wants them committed separately: stage and commit each logical change on its own (`git add <paths>` per group, one commit each) with a distinct message per commit. Don't lump unrelated changes into one commit.
- If the user hasn't said whether to split, and the working tree mixes unrelated changes, ask whether to split or commit together before committing.
